# Vibe-coding — Part 2 (livestream)

> Original: "Вайб-кодінг - частина 2" — https://youtu.be/LtaOkeyq6Bc

Continuation of the racing-telemetry vibe-coding stream, which turns into a rich Q&A. The author first demos an earlier proof-of-concept he built in half a day with Google's Lit web components: it loads a 25 Hz GPS telemetry CSV, draws laps on a map, and lets him compare lap vs lap with a delta graph — he experimented with signal reconstruction via linear interpolation, splines and a Kalman filter. Then he vibe-codes the "garage" (vehicles CRUD) endpoints with [[claude-code|Claude Code]] from the API docs Claude generated in part 1 — and spends much of the time catching the AI's mistakes: it duplicated types instead of using [[livr|LIVR]] type inference (he writes the `InferInterface<typeof validationRules>` example by hand to teach it), and it produced inconsistent `status`/`success` fields because the docs themselves were inconsistent. His thesis: **the better structured your project, the better the AI works** — after a few weeks of iteratively encoding corrections into `CLAUDE.md`, every new feature "just lands". The stream closes with a deep-dive into his LIVR validator's design doc versus Zod, and war stories from his pre-cloud backend days.

## Key takeaways
- **AI is a junior, and it scales like juniors** ([[ai-and-jobs]]): with one junior it's often faster to do it yourself; with 10 juniors arriving monthly you invest in style guides, static analysis and standards — same with AI agents. To the viral "$200/month Opus 4.5 saves $4000 on a designer and a middle dev" screenshot he quotes the rebuttal: "bought a $300 multicooker, saving $3000 on a chef — multicookers are taking over cooking". The dev remains the primary consumer of the tool; "10 juniors without a senior accomplish nothing", so AI-generated code is not free — someone must read all of it.
- **AI over-engineers** ([[abstractions]]): it invents needless abstractions — e.g. a brand-manager class hierarchy with `registerNewBrand()` where a plain JSON file of car brands would do. He constantly says "delete this, delete that abstraction". This is one core reason you must read every line, like a human PR ([[code-review]]) — "AI already reviews PRs more than it writes them, but a human still reviews what's left".
- **Claude Code workflow tips:** `/clear` before every task (he uses the newer "clear context and accept edits" option so the approved plan becomes a fresh prompt); run several Claudes in parallel using **git worktrees** — one local repo, many working directories, which Anthropic's own docs recommend; feed docs directly (or context7) when the model's knowledge is stale. Fun fact he loves: Claude Code, Gemini CLI and Codex CLIs are all written in React. He tried [[cursor|Cursor]] and it "didn't do what I needed"; a CLI he can launch anywhere in his Quake-style terminal beats installing a separate editor.
- **Prompting in Ukrainian works fine:** he explains it via word2vec/embeddings — semantically close words across languages sit near each other in the shared vector space, so the model has one multilingual "core". (Local models via Cline, ≤14B params, were "trash" by comparison.)
- **Testing in the AI era** ([[automated-testing]]): unit tests live next to the files, integration/API tests separately; parallelize DB-bound tests by spinning one container with 10–20 databases and mapping test-worker ID → database number. Tests are now cheap because the AI writes them — and the AI *needs* them (plus TypeScript/static analysis) as its feedback loop, since it won't click through your UI. Whether frontend tests are worth it "depends": in Google's monorepo a shared Angular can update under you so everything must be tested; Instagram famously shipped its React web app with zero frontend tests.
- **LIVR vs Zod deep-dive:** LIVR rules are pure data (serializable, transformable — you could compile them to Zod, not vice versa), support DDD-flavored rule aliasing (`required + positive_decimal` aliased as `valid_price` with its own `INVALID_PRICE` error code), and do no code generation (a security stance — he calls eval-style validators "JS execution"). Warm performance is comparable (Zod ~4M ops/s vs LIVR ~3.4M), but with dynamically constructed rules Zod is ~94× slower.
- **Performance is architecture, and humans hold the context** ([[software-design]]): he *knows* there will only be ~100–200 race tracks (cache them in memory forever, no pagination) but millions of telemetry datapoints (never store one entity per point) — the AI doesn't know that and will "decide something for itself that bites you later". Supporting stories: Figma obsessed over optimization from day one (Rust→WASM modules), and VS Code's Monaco editor deliberately rejected all web frameworks for a specialized text-rendering engine — while Atom, which didn't, lost.
- **Plans:** extract the emerging backend into an opinionated boilerplate (his `chista`/`chista-express` modules, a refresh of WebbyLab's 6-year-old starter kit) with `CLAUDE.md` and skills included, "so the AI already knows how to work with it" — setting up infrastructure is exactly the complex, decision-heavy task a junior/AI can't do alone. Also: vibe-code a site for WebbyLab's charity fund (the company donates 50% of profit), and maybe a game — he loved [[godot|Godot]]'s simplicity, and already vibe-coded a mini top-down shooter in JS with his son in a couple of hours.
- **War story** ([[career-and-growth]]): his first frontend was Mail.ua — a single-page app for IE6 — while writing Perl backends with their own framework, ZFS storage and diskless PXE-booted frontend servers that pulled a Linux image into RAM on reset. "Everything the cloud does now, we did by hand."
